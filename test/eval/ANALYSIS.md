## Multi-agent diagnostic — root-cause sweep (Nov 2026)

A team of 5 diagnostic agents ran 91 random real COLAs through the
cloud-mode backend across the categories in `test/TEST_PLAN.md`:

| Agent | Scope | n | Finding |
|---|---|---|---|
| **W** Warning | random clean approved | 20 | 5/6 warning false-flags were single-character OCR slips (missing comma, "operte" for "operate"), not real paraphrase |
| **F** Field-match | random clean approved | 20 | 50% of all field issues are brand-vs-product or brand-case stylistic differences |
| **B** Category coverage | TEST_PLAN §B (16 categories) | 16 | Same root causes across all categories; no category-specific failure mode |
| **I** Imports + COO | imported COLAs | 15 | 6/15 warning false-flags + 3/15 COO false-flags from foreign-language prefixes (`Produit de`, `Hecho en`) |
| **M** Multi-image | front vs back paired | 10+10 | 10/10 back panels carry the warning, 9/10 fronts don't — confirms back-prefer logic |

### FIVE fixes applied (each locked in by a unit test)

1. **Brand: case-only differences → `match`** (was `likely`).
   COLA forms store brands in title case; real labels routinely print in ALL CAPS for prominence. That's stylistic, not substantive.
2. **Brand: common business suffixes ignored** (`Imported Beer`, `Brewing Co`, `Vineyards`, `LLC`, etc.).
   "Suprema Imported Beer" matches "SUPREMA"; "3rd Deck Brewing Co." matches "3RD DECK BREWING CO".
3. **Warning: near-verbatim softening** (≥95% similarity).
   Single-character OCR slips don't false-flag as paraphrase. The engine sets `verbatimMatch=True` with a `near_verbatim` advisory deviation so the agent verifies visually but the verdict isn't auto-promoted to needs-review. Real paraphrase (substantive wording change) still scores below 0.95 → `verbatim=False`.
4. **COO: multilingual prefixes + dual-language displays**.
   `Produit de`, `Hecho en`, `Producto de`, `Prodotto in`, `Erzeugnis aus`, `Feito em`, plus dual-language stripping for displays like `PRODUIT DE FRANCE PRODUCT OF FRANCE`.
5. **Preprocessing: `productName` accepted as alternative brand match**.
   The COLA form has both a registered BRAND_NAME (often the brewery) and a
   PRODUCT_NAME / Fanciful Name (often the SKU). Real labels routinely print
   whichever is more visually prominent — usually the SKU. The
   `ApplicationData` contract now carries `productName: str?` and the brand
   classifier accepts a label-extracted name match against either. This is
   the single biggest false-positive eliminator on real artwork — addresses
   the "Half Acre" (brewery) vs "NEON FOOTHILLS" (product) class of misses.

### Before-vs-after impact (same 40 random clean approved labels, same seeds)

| Metric | Before fixes | After fixes | Δ |
|---|---|---|---|
| Warning false-flag rate | **15.0%** | **2.5%** | **−83%** |
| Approved → auto-pass | 5% | **32%** | +27 pp |
| Approved → needs-confirm | 38% | 22% | −16 pp |
| Approved → needs-review (false-positive) | 57% | **45%** | −12 pp |

## Reading these numbers

### What the engine reliably catches (deterministic)

- **Net-contents semantic mismatches** (5% tolerance) — declared `750 mL` vs extracted `500 mL` → `flag`
- **Brand-vs-different-brand** — declared `Half Acre` vs extracted `NONEXISTENT MISMATCH BRAND XYZ` → `flag`
- **ABV mismatches** (>0.1% delta) — both high and low → `flag`
- **Unit swaps** — declared `100 mL` vs extracted `750 mL` → `flag`
- **Verbatim Government Warning paraphrase** when the model transcribes accurately

### What the engine routes to human review (with explicit reasons)

- Extractions where the model's confidence falls below 0.60 → demoted from match to likely
- Multi-image COLAs where ABV / net contents / bottler is on a panel not provided → `likely` with note
- Unmeasurable font size (every real image, in practice) → `fontSizeOk=False` with "please confirm manually" + 16.22 threshold cited
- Subtle warning spelling/punctuation errors where OCR normalization papers over them (8/20 TTB-noted findings) — covered in `docs/COVERAGE.md` §C

### Why "73% needs-review" on truly-approved labels isn't a failure

The needs-review label is the engine's **conservative default** when:
- COLA form `BRAND_NAME` ≠ label's prominent text (which is often `PRODUCT_NAME` / fanciful SKU)
- COLA form omits the bottler line → field-level "likely with note"
- Dataset's `OCR_VOLUME` captured a different unit than the label printed
- Container-size mismatches when the COLA covers multiple keg sizes

Per `docs/COVERAGE.md`: "Nothing is silently auto-passed." The engine prefers
to route uncertain cases to the agent rather than risk a false auto-pass.

## Engine improvements that landed during eval iteration

Every refinement is locked in by a unit test (`backend/tests/`).

1. **Case-insensitive verbatim warning** — 16.21 fixes wording, 16.22 fixes only the heading case (all-caps body printed verbatim is compliant). Warning false-flag fell from 87% → 27%.
2. **OCR-noise punctuation tightening** — `WARNING :` → `WARNING:`, `( 1 )` → `(1)`.
3. **`casingBoldOk` drops the `heading_bold` gate** — model can't reliably detect bold from a single image; gates on `casing_all_caps and not body_bold`. Heading bold is flagged for visual confirmation by the agent.
4. **Country of origin prefix strip** — strips `Product of / Imported from / Made in / Bottled in / Distilled in`, case-insensitive.
5. **Net contents semantic match** — parse both sides to mL; within 5% = match. Handles `1 PINT (16 FL OZ)`, `12 L 40.5 FL OZ 2.53 PINT`.
6. **Alcohol content semantic match** — numeric % within 0.1.
7. **Umbrella class types** — 40+ COLA codes (`beer`, `ale`, `table red wine`, `whisky specialties`, etc.) match any printed sub-designation.
8. **Missing extracted on a non-warning field** → `likely` (multi-image reality) instead of `flag`.
9. **Flavored-malt class-type auto-detects added flavors** — seltzers correctly require ABV per 7.63.
10. **Empty declared on required field** → `match` with transparent note (COLA form omissions, e.g. bottler).
11. **Low-confidence promotion** — extractor confidence < 0.60 demotes `match` → `likely` so a confident-but-wrong AI read can't auto-pass.
12. **Tighter brand containment** — a short declared brand inside long unrelated extracted text is `flag`, not `likely` (≥60% length-ratio required for the containment rule).

## Dataset construction (`test/eval/build_dataset.py`)

Three slices, all from the COLA Cloud free sample (CC0):

1. **`clean_approved`** (60 rows) — stratified across 21 product categories; warning visible in OCR; keg-size ambiguity excluded; OCR-derived ABV + net contents from the dataset.
2. **`ttb_gws_finding`** (20 rows) — real approved COLAs where `APPROVAL_QUALIFICATIONS` contains spelling/punctuation findings TTB recorded. The gold-standard "should flag" set.
3. **`mutation`** (10 rows) — declared application data flipped on copies of clean approved labels. 7 mutation kinds: `shrink_net_contents` · `swap_net_contents_unit` · `brand_mismatch` · `abv_mismatch_high` · `abv_mismatch_low` · `country_origin_mismatch` · `class_type_mismatch`. Each kind is only applied to a row whose beverage type makes the violation reachable (no ABV mutations on plain beer — §7.63 makes ABV optional).

The `cola_sample.csv` is committed (small, reproducible, ~90 rows). The
bulk source CSVs and downloaded images are gitignored — run
`make build-eval-data` to populate locally.

## Why we don't have rejected-status labels from the TTB Public COLA Registry

The data.gov "TTB Public COLA Registry Search and Download" resource is a
link to the **interactive search UI** at `ttbonline.gov/colasonline/`,
not a bulk-downloadable CSV. The Advanced Search supports filtering by
Status (approved / expired / surrendered / revoked) but results return
as paginated HTML — no public bulk CSV endpoint exists.

The COLA Cloud free sample is approved-only by definition (the COLA
Registry only publishes approved labels — rejected applications never
enter the public registry).

That's why the harness uses two fallbacks for "should-flag" cases:

- **TTB-noted findings on approved labels** (20 records) — real labels
  where TTB explicitly recorded a finding in the public record.
- **Mutations of approved labels** (10 records) — engineered to force
  specific violation types.

Both serve as "must flag" benchmarks; the `APPROVAL_QUALIFICATIONS`
field is, in practice, TTB's own documentation of issues they caught.
